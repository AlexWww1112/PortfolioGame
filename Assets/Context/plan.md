# Goal

围绕“尺寸变化会改变物体属性”的核心机制，做一个可玩、可验证、可扩展的解谜冒险原型。  
首个里程碑不是堆功能，而是先证明一条核心体验链路成立：

`观察尺寸线索 -> 改变尺寸 -> 属性随之变化 -> 世界反馈变化 -> 解开谜题`

# Design Pillars

1. 尺寸不是单纯视觉效果，而是统一驱动重量、伤害、音量、容量、交互条件等规则。
2. 同一条规则要能复用于玩家、道具、机关、NPC，避免每个谜题单独写例外逻辑。
3. 谜题优先强调“理解规则并组合使用”，而不是纯操作难度。
4. 每个关卡至少让玩家看到一次“变大”和“变小”各自的价值，避免玩法单向度。

# Core System Decisions

## 1. 尺寸规则模型

- 建议原型阶段使用“离散尺寸档位”，不要做连续无级缩放。
- 建议档位：`XS / S / M / L / XL`
- 建议倍率：`0.5x / 0.75x / 1x / 1.5x / 2x`
- 选择离散档位的原因：
  - 更容易做碰撞、交互范围、动画和关卡校准
  - 更容易给玩家明确反馈
  - 后续如果验证手感足够好，再考虑扩展为连续缩放

## 2. 属性联动规则

- 重量：尺寸越大越重，影响压板、被风吹动、是否能被玩家搬运
- 伤害：尺寸越大基础伤害越高，决定炸弹破墙、武器砍断障碍的阈值
- 音量：尺寸越大音量越高，可用于 NPC 对话、远距离提示、字幕大小联动
- 交互门槛：只有达到合适尺寸才能插入、放入、拿取或触发机关
- 容量：背包尺寸越大，可容纳的格数或总体积越大
- 时间流速：将倒计时/时钟相关对象放大时，时间流速减慢；缩小时变快

## 3. 原型阶段的约束

- 所有“可缩放对象”都应走同一套尺寸状态系统
- 每个可破坏物必须明确“所需伤害阈值”
- 每个可交互插槽必须明确“允许尺寸范围”
- 时间类机关先只做一个全局演示版本，避免早期系统过散

# Prototype Scope

## 核心能力

1. 选中可缩放对象并切换尺寸档位
2. 尺寸变化后，同步影响至少三类属性：伤害、交互条件、音量/文本表现
3. 玩家可对自身进行缩放
4. 基础关卡支持尺寸解谜闭环

## 首批玩法模块

### A. 破坏型谜题

- 炸弹变大后可炸开墙体
- 剑或斧头变大后可劈开灌木或木障碍
- 目标：验证“尺寸 -> 伤害阈值 -> 世界变化”

### B. 尺寸门槛交互

- 钥匙、齿轮、方块等物体只有在合适尺寸时才能放入插槽或被取出
- 目标：验证“尺寸 -> 是否允许交互”

### C. 玩家自身缩放

- 玩家可变大或变小以通过不同区域
- 镜子中可观察并选取“自己”的映像作为交互媒介
- 目标：验证自我缩放与观察反馈是否有独特价值

### D. 时间机关

- 倒计时数字或时钟可被缩放
- 变大时倒计时变慢，给玩家争取行动窗口
- 目标：验证“缩放对象本身”也能改变抽象系统，不只影响实体物理

### E. 背包容量变化

- 背包变大时可携带更多关键道具
- 谜题可要求先扩容，再搬运多个组件
- 目标：验证“尺寸改变资源管理”

### F. 创造平台

- 将小物体放大形成垫脚平台
- 或将现有平台缩小后带走，再在别处放大使用
- 目标：验证“尺寸改变空间路径”

### G. NPC 反馈联动

- NPC 尺寸变化时，语音音量、字幕大小、存在感同步变化
- 可作为情绪演出，也可作为提示强弱反馈
- 目标：验证该机制在叙事与反馈层也成立

# Vertical Slice Proposal

建议先做一个 10 到 15 分钟的单关卡纵切，而不是并行开发全部系统。

## 推荐关卡顺序

1. 教学房：学会选中物体并切换尺寸
2. 破墙房：放大炸弹或武器，理解伤害阈值
3. 钥匙房：将钥匙调到合适尺寸才能开门
4. 平台房：放大箱子或石块创造路径
5. 倒计时房：放大时钟让门保持开启更久
6. 收尾房：组合使用前面 2 到 3 条规则

## 暂不进入纵切的内容

- 大量 NPC 叙事分支
- 复杂背包 UI
- 连续缩放
- 高自由度物理解法

# Implementation Phases

## Phase 1. 规则落地

- 定义尺寸档位、全局倍率表、属性映射关系
- 明确哪些对象支持缩放，哪些对象只响应缩放结果
- 产出一份统一规则表，避免后续关卡各写各的

## Phase 2. 最小交互闭环

- 完成对象选取、缩放切换、反馈表现
- 完成一个“尺寸影响伤害”的破坏谜题
- 完成一个“尺寸影响交互条件”的插槽谜题

## Phase 3. 玩家与空间

- 支持玩家自身缩放
- 支持平台创造与路径改变
- 验证角色控制、相机、碰撞体在不同尺寸下是否稳定

## Phase 4. 抽象系统联动

- 接入时间机关
- 接入背包容量变化
- 补充 HUD 或场景反馈，确保规则容易被理解

## Phase 5. 演出与扩展

- 接入 NPC 尺寸反馈
- 微调音频、字幕、镜子演出
- 评估是否值得继续扩展到更多属性

# Next Steps

1. 先把“尺寸档位 + 属性映射 + 可缩放对象判定”整理成具体规则表。
2. 选三个最能证明玩法成立的首发模块：`破坏型谜题`、`尺寸门槛交互`、`创造平台`。
3. 在此基础上搭一个单关卡纵切，再决定是否加入 `玩家自身缩放` 和 `时间机关`。
4. 若纵切验证通过，再扩展 `背包容量` 与 `NPC 反馈`，避免前期系统发散。

# Open Questions

- 镜子交互是“选中镜中自己来改变现实尺寸”，还是“镜像世界中存在可独立操控的映像”？
- 背包容量是用“格子数”表达，还是用“总体积上限”表达？
- 时间流速变化是只影响特定机关，还是会影响整个房间？
- 敌人是否会参与尺寸规则，还是原型阶段仅服务于解谜？

# Deferred

- 连续缩放与物理精细联动
- 多种武器各自独立成长曲线
- 更复杂的 NPC 社交或剧情分支
- 大规模开放式关卡
