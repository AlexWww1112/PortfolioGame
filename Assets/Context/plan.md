# Goal

以 Meta Quest 3 原生体验为主线，构建“尺寸变化会影响物体其他属性”的游戏功能框架，并在架构上预留同一套核心玩法向非 VR 版本转换的接口和空间。

Codex 的职责是工程实现与组件边界设计，不负责决定系统规则、数值平衡、关卡结构或谜题设计。所有系统设计和关卡设计均由用户手动完成。

# Responsibility Boundary

## User Owns

- 系统规则设计
- 关卡设计
- 谜题目标与解法
- 尺寸倍率限制
- 属性映射公式或数值表
- 每个物体、机关、NPC 的具体玩法用途
- 交互条件、破坏阈值、背包容量、时间流速等具体设计参数
- VR 与非 VR 的最终交互手感取舍
- 将脚本接入 Unity

## Codex Owns

- 按用户提供的规则实现可运行功能
- 搭建可复用、可配置的 Unity 脚本结构
- 保持核心玩法逻辑平台无关
- 为 Meta Quest 3 原生交互提供优先支持
- 为非 VR 版本预留输入、射线来源、反馈与移动方式的替换接口
- 在实现后更新 `state.md`、`code_map.md` 和 `memory.md`

# Implementation Principles

- Meta Quest 3 是主目标平台，后续 PC 或普通手柄版本属于同一套核心玩法的适配层，而不是独立重写版本。
- 核心系统不直接依赖鼠标、键盘、头显、手柄模型、UI 准星或特定相机实现。
- 输入、选择射线、视觉反馈、玩家移动应与核心玩法逻辑分层。
- 不主动替用户补充玩法设计，只实现用户已明确的系统需求。
- 所有尺寸相关规则应尽量数据化或 Inspector 可配置，方便用户手动调参。
- 通用系统只表达机制能力，不写死特定关卡解法。
- 新增代码文件前需要先获得用户许可。
- 所有脚本由用户手动 attach 到场景对象上，代码中不要生成任何 objects。
- 尽量少写兜底机制；缺少必要引用时应明确报错并停止该脚本，而不是自动查找或猜测替代对象。
- 优先复用 Unity 现有组件和项目已有代码，避免过度设计。

# Target Architecture

## 1. Platform-Agnostic Gameplay Core

这部分应尽量同时服务于 VR 与非 VR：

- 可交互对象状态
- 尺寸变化与尺寸限制
- 物体被选中、被拿起、被放下的状态
- 尺寸变化触发的属性联动
- 尺寸门槛判定
- 破坏判定
- 玩家尺寸状态
- 时间、背包、平台、NPC 等系统的规则执行

## 2. Meta Quest 3 Native Layer

这部分优先按 Quest 3 体验实现：

- XR 输入动作映射
- XR 射线来源或手部交互来源
- XR 玩家移动与转向
- XR 版本的交互反馈
- XR 相机 / rig 相关适配

## 3. Non-VR Adapter Layer

这部分作为后续可接入层预留：

- 键鼠输入适配
- 普通手柄输入适配
- 非 VR 的中心射线或其他选择方式
- 非 VR 的屏幕反馈与 UI
- 非 VR 的第一人称或其他移动实现

# Feature Backlog

以下内容是待实现功能方向，不代表具体系统规则已确定。实际实现前需要以用户提供的设计参数为准。

## 0. 输入与交互基础

- 使用 Unity Input System 作为统一输入基础。
- 优先定义适用于 Quest 3 的动作集合，例如移动、转向、选择、缩放、抓取、确认、取消。
- 输入读取层应允许后续接入键鼠和普通手柄，而不改写核心玩法逻辑。
- 选择逻辑应预留可替换的射线来源，而不是写死某个相机实现。

## 1. 物体选取、拿起、放置与缩放

- 支持玩家选中可交互物体。
- 支持玩家拿起和放下物体。
- 支持在拿起状态下改变物体大小。
- 支持 `IsSelected`、`IsHeld`、当前尺寸倍率等核心状态。
- 该逻辑应能被 Quest 3 的 XR 交互方式直接驱动，也能被未来非 VR 适配层复用。

## 2. 尺寸变化基础系统

- 支持物体尺寸变化。
- 支持尺寸上下限或尺寸档位。
- 支持尺寸变化后通知其他组件刷新属性。
- 支持在 Inspector 中配置对象是否可被缩放、默认尺寸、最小尺寸、最大尺寸等参数。

## 3. 属性联动系统

- 尺寸变化可影响重量。
- 尺寸变化可影响伤害。
- 尺寸变化可影响音量。
- 尺寸变化可影响字幕或 UI 显示大小。
- 尺寸变化可影响背包容量。
- 尺寸变化可影响时间流速。
- 具体映射关系由用户设计，代码只提供可配置的执行能力。

## 4. 破坏与工具交互

- 支持炸弹破墙。
- 支持剑或斧头破坏指定障碍，例如灌木。
- 支持尺寸影响破坏判定。
- 破坏所需条件、目标类型和阈值由用户配置。

## 5. 尺寸门槛交互

- 支持物体达到指定尺寸范围后才能交互。
- 支持钥匙等对象在合适尺寸下触发指定行为。
- 支持物体在被拿起状态下与门等目标对象交互。
- 支持交互失败时返回明确结果，便于用户接入反馈表现。

## 6. 玩家自身尺寸变化

- 支持改变玩家自身尺寸。
- 玩家尺寸状态本身应独立于具体平台。
- Quest 3 与非 VR 对玩家尺寸的应用方式允许分开实现。
- 镜子选取与交互逻辑按用户后续设计实现。

## 7. 时间相关交互

- 支持倒计时数字或时钟对象被缩放。
- 支持尺寸影响时间流速。
- 作用范围、倍率关系和触发条件由用户设计。

## 8. 背包容量变化

- 支持背包尺寸影响容量。
- 容量表达方式由用户决定，例如格子数、总体积或其他规则。
- 代码应提供清晰接口，方便关卡或 UI 读取当前容量。

## 9. 平台创建或路径改变

- 支持物体尺寸变化后改变可站立、可碰撞或可通行状态。
- 是否作为平台、如何放置、是否可搬运由用户设计。

## 10. NPC 对话反馈

- 支持 NPC 尺寸影响音量。
- 支持 NPC 尺寸影响字幕大小。
- 具体对话内容、演出规则和触发方式由用户设计。

# Implementation Phases

## Phase 1. 统一方向并收敛到 Quest 3 原生主线

- 明确后续实现以 Meta Quest 3 为主目标。
- 收敛当前并存的旧输入栈与新输入栈，避免继续扩大分叉。
- 明确哪些旧脚本只作为过渡原型，哪些需要保留并演化。
- 确认后续核心交互动作命名与输入边界。

## Phase 2. 建立平台无关的核心交互状态

- 实现或重构可交互对象的核心状态。
- 让选中、拿起、放下、缩放成为独立于平台输入的通用逻辑。
- 为尺寸事件、属性刷新、交互结果提供清晰接口。

## Phase 3. 建立 Quest 3 原生交互层

- 接入适合 Quest 3 的输入动作。
- 实现 XR 版本的选择来源、拿取交互和反馈方式。
- 实现 XR 版本的移动与转向方式。
- 确保 Quest 3 交互层调用的是同一套核心玩法接口。
- Quest bridge 骨架应先通过 UnityEvent 与 Meta Building Blocks 接通，再决定是否需要更深层的 SDK 绑定。

## Phase 4. 实现尺寸驱动的核心玩法系统

- 按用户规格实现重量、伤害、音量、时间、容量等联动。
- 实现破坏判定、尺寸门槛交互与对象间交互。
- 实现玩家尺寸状态及其对其他系统的影响。

## Phase 5. 预留并接入非 VR 适配层

- 在不改写核心玩法逻辑的前提下，接入键鼠或普通手柄版本。
- 为非 VR 版本实现独立的输入、射线、反馈和移动适配。
- 验证 Quest 3 与非 VR 是否真正共用同一套核心玩法逻辑。

## Phase 6. 集成、修复与迭代

- 修复实现层 bug。
- 根据用户反馈调整 Inspector 参数和组件边界。
- 处理 Quest 3 实机体验中暴露的问题。

# Next Steps

1. `Scene1` 当前阶段按现有目标暂时视为完成，后续若关卡需求变化再回补。
2. 实现“尺寸变化驱动其他属性变化”的通用联动能力，优先支持重量、伤害、音量、容量、时间倍率等可配置数值输出。
3. 保持属性联动层平台无关，不把具体规则写死在 Quest bridge、`ObjectSelector` 或单一关卡对象里。
4. 在 Unity 场景里验证 `SceneSpawnPoint + GameManager` 的 VR 场景切换出生点对齐，确认 `Scene1 -> Scene2` 不再沿用旧追踪偏移落到错误位置。
5. 在完成属性联动基础后，再根据用户提供的具体对象规则继续接入破坏判定、玩家尺寸、时间对象、背包等系统。
6. 每完成一个 non-trivial 功能，更新 `state.md`、必要时更新 `code_map.md` 和 `memory.md`。

# Deferred

- 未经用户明确设计的规则推导。
- 未经用户要求的关卡方案。
- 未经用户要求的数值平衡建议。
- 未经用户许可的新代码文件创建。
- Quest 3 之外的其他 XR 平台专项适配。
- 非 VR 版本的最终表现和手感打磨。
