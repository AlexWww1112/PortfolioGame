# Goal

根据用户手动设计的系统，实现“尺寸变化会影响物体其他属性”的游戏功能框架。

Codex 的职责是工程实现，不负责决定系统规则、数值平衡、关卡结构或谜题设计。所有系统设计和关卡设计均由用户手动完成。

# Responsibility Boundary

## User Owns

- 系统规则设计
- 关卡设计
- 谜题目标与解法
- 尺寸倍率限制
- 属性映射公式或数值表
- 每个物体、机关、NPC 的具体玩法用途
- 交互条件、破坏阈值、背包容量、时间流速等具体设计参数
- 将脚本接入Unity

## Codex Owns

- 按用户提供的规则实现可运行功能
- 搭建可复用、可配置的 Unity 脚本结构
- 保持系统边界清晰，避免把关卡逻辑硬编码进通用系统
- 提供必要的 Inspector 参数、事件接口和组件组合方式
- 在实现后更新 `state.md`、`code_map.md` 和 `memory.md`

# Implementation Principles

- 不主动替用户补充玩法设计，只实现用户已明确的系统需求。
- 所有尺寸相关规则应尽量数据化或 Inspector 可配置，方便用户手动调参。
- 通用系统只表达机制能力，不写死特定关卡解法。
- 边界条件应自然融入通用逻辑，例如尺寸限制、交互失败、属性刷新、对象缺失等。
- 新增代码文件前需要先获得用户许可。
- 所有脚本由用户手动 attach 到场景对象上，代码中不要生成任何 objects。
- 优先复用 Unity 现有组件和项目已有代码，避免过度设计。

# Feature Backlog

以下内容是待实现功能方向，不代表具体系统规则已确定。实际实现前需要以用户提供的设计参数为准。

## 1. 尺寸变化基础系统

- 支持物体尺寸变化。
- 支持尺寸上下限或尺寸档位。
- 支持尺寸变化后通知其他组件刷新属性。
- 支持在 Inspector 中配置对象是否可被缩放、默认尺寸、最小尺寸、最大尺寸等参数。

## 2. 属性联动系统

- 尺寸变化可影响重量。
- 尺寸变化可影响伤害。
- 尺寸变化可影响音量。
- 尺寸变化可影响字幕或 UI 显示大小。
- 尺寸变化可影响背包容量。
- 尺寸变化可影响时间流速。
- 具体映射关系由用户设计，代码只提供可配置的执行能力。

## 3. 破坏与工具交互

- 支持炸弹破墙。
- 支持剑或斧头破坏指定障碍，例如灌木。
- 支持尺寸影响破坏判定。
- 破坏所需条件、目标类型和阈值由用户配置。

## 4. 尺寸门槛交互

- 支持物体达到指定尺寸范围后才能交互。
- 支持钥匙等对象在合适尺寸下触发指定行为。
- 支持交互失败时返回明确结果，便于用户接入反馈表现。

## 5. 玩家自身尺寸变化

- 支持改变玩家自身尺寸。
- 玩家尺寸变化后应同步处理碰撞、移动、相机或其他受影响组件。
- 镜子选取与交互逻辑按用户后续设计实现。

## 6. 时间相关交互

- 支持倒计时数字或时钟对象被缩放。
- 支持尺寸影响时间流速。
- 作用范围、倍率关系和触发条件由用户设计。

## 7. 背包容量变化

- 支持背包尺寸影响容量。
- 容量表达方式由用户决定，例如格子数、总体积或其他规则。
- 代码应提供清晰接口，方便关卡或 UI 读取当前容量。

## 8. 平台创建或路径改变

- 支持物体尺寸变化后改变可站立、可碰撞或可通行状态。
- 是否作为平台、如何放置、是否可搬运由用户设计。

## 9. NPC 对话反馈

- 支持 NPC 尺寸影响音量。
- 支持 NPC 尺寸影响字幕大小。
- 具体对话内容、演出规则和触发方式由用户设计。

# Implementation Phases

## Phase 1. 等待并确认用户系统规格

- 确认尺寸系统使用连续缩放还是离散档位。
- 确认尺寸上下限与默认值。
- 确认属性联动采用曲线、公式、表格还是手动配置。
- 确认首批需要实现的功能模块。

## Phase 2. 实现尺寸基础能力

- 实现可缩放对象的通用组件。
- 实现尺寸限制。
- 实现尺寸变化事件。
- 实现属性刷新入口。

## Phase 3. 实现属性联动组件

- 按用户确认的规则实现重量、伤害、音量、字幕、容量、时间等联动。
- 每个联动能力应能独立启用或关闭。
- 避免将某个关卡的特殊逻辑写入通用组件。

## Phase 4. 实现交互与判定组件

- 实现破坏判定。
- 实现尺寸门槛交互。
- 实现玩家自身尺寸变化。
- 实现背包、时间、平台、NPC 等模块中用户优先级最高的部分。

## Phase 5. 接入用户关卡内容

- 修复实现层 bug。
- 根据用户反馈调整接口、Inspector 参数和组件边界。

# Next Steps

1. 用户先确认首批要实现的功能模块。
2. 用户提供对应系统规则或最小可用参数。
3. Codex 在获得新增脚本许可后，实现第一批 Unity 组件。
4. 每完成一个非 trivial 功能，更新 `state.md`、必要时更新 `code_map.md` 和 `memory.md`。

# Deferred

- 未经用户明确设计的规则推导。
- 未经用户要求的关卡方案。
- 未经用户要求的数值平衡建议。
- 未经用户许可的新代码文件创建。
